网络地址转换与穿越
[ Network ]

Posted on 8/18/2021 Edited on 9/20/2021 Symbols count in article: 20k Reading time ≈ 18 mins.

Network address translation (NAT) 是指当数据包经过路由设备时，通过修改IP包头中的网络地址信息，将一个IP地址空间映射到另一个IP地址空间的方法。最初用于更换上游 Internet service provider 时，避免重新分配网络中每个主机的地址，后来在面对IPv4地址耗尽问题时变成一种流行而必要的工具。

私网地址

RFC1918 Address Allocation for Private Internets (1996)

The Internet Assigned Numbers Authority (IANA) has reserved the
following three blocks of the IP address space for private internets:

 10.0.0.0        -   10.255.255.255  (10/8 prefix)
 172.16.0.0      -   172.31.255.255  (172.16/12 prefix)
 192.168.0.0     -   192.168.255.255 (192.168/16 prefix)

NAT应用场景

  • 需要连接到因特网，但主机没有全局唯一的IP地址；
  • 更换的ISP要求对网络进行重新编址；
  • 需要合并两个使用相同编址方案的内联网；
优点 缺点
节省合法的注册地址 地址转换将增加交换延迟
在地址重叠或网络发生变化时避免重新编址 导致无法进行端到端的IP跟踪
提高连接到因特网的灵活性 导致有些应用程序无法正常运行

NAT设备特征

  • 透明的地址分配：
    • 静态地址分配：在私网地址和外部地址之间进行一对一的映射。使用时必须为私网中的每台主机提供一个外部IP地址；
    • 动态地址分配；将私网地址动态映射到外部地址池中的一个地址。使用时无需静态的配置路由器，但必须有足够的公网IP地址，让连接到因特网的主机都能够同时发送和接收分组。
  • 通过地址转换实现透明路由（只转发包，不交换路由信息），分为三个阶段：
    1. 地址绑定：在新会话的开始分配地址，并创建一个状态，以促进与会话相关的数据报的转换。
    2. 地址查找与转换：一旦为会话建立了状态，属于该会话的所有包都将进行地址查找和转换。
    3. 地址解绑：当NAT设备认为使用地址绑定的最后一个会话已经终止时，它将执行地址解除绑定。
  • ICMP错误数据包载荷转换：为了让NAT设备对终端主机完全透明，IP、传输层以及ICMP包头中的地址和校验和都必须被修改。

NAT变体

RFC2663 IP Network Address Translator (NAT) Terminology and Considerations (1999)

名称 特征
Traditional NAT / Outbound NAT 会话只允许从内部主机发起。
Basic NAT / one-to-one NAT 只转换 IP、UDP、TCP、ICMP 包头中的 IP 地址。
Network Address Port Translation (NAPT) / one-to-many NAT / Port Address Translation (PAT) / IP masquerading / NAT overload 进一步扩展了转换的概念，例如 TCP 和 UDP 端口、ICMP 查询标识符等也被转换。
Two-Way NAT / Bi-directional NAT 会话既能从内部主机发起，也能从外部主机发起。
Twice NAT 数据报跨越地址域时，源地址和目的地址都会被修改，应用于内部地址和外部地址有冲突的情况下。

NAPT

RFC3498 STUN - Simple Traversal of User Datagram Protocol (UDP) Through Network Address Translators (NATs) (2003)
RFC4787 Network Address Translation (NAT) Behavioral Requirements for Unicast UDP (2007)

NAPT实现分类

  • RFC3498 (分类部分定义不够清晰，已弃用)：
    • 锥型（Cone）:
      • 全锥型（Full Cone）：
        内部主机向外打了一个洞，外网的任何主机都可以利用这个洞与它通信。
      • 限制锥型（Restricted Cone）：
        内部主机向某一外部主机打了一个洞后，只有该外部主机才能利用这个洞。
      • 端口限制锥型（Port Restricted Cone）：
        内部主机向外部主机上的某一程序（端口）打了一个洞，则只有该程序可以利用这个洞，其他的不行。
    • 对称型（Symmetric）：
      对称型NAT相当于对同一内部主机联系不同的外部主机时都需要打不同的洞。
  • RFC4787：
    • 地址和端口映射（Address and Port Mapping）：从内部主机发送包时。
      • Endpoint-Independent Mapping：
        从内部主机发出的包，都会重用已分配的外部地址。例如 Full-Cone, Restricted-Cone, and Port-Restricted Cone NATs。
      • Address-Dependent Mapping：
        当内部主机发出的包中目的IP不变时，已分配的外部地址才会重用。例如 Symmetric NAT。
      • Address and Port-Dependent Mapping：
        当内部主机发出的包中目的IP&port不变时，已分配的外部地址才会重用。例如 Symmetric NAT。
    • 过滤行为（Filtering Behavior）：外部主机应答包回来时。
      • Endpoint-Independent Filtering：
        外部地址上收到的任何包，都会被转发到内部主机上。例如 Full-Cone NAT。
      • Address-Dependent Filtering：
        当外部地址曾经向应答包中源IP发送过数据，应答包才会被转发到内部主机上。Restricted-Cone NAT, Symmetric NAT。
      • Address and Port-Dependent Filtering：
        当外部地址曾经向应答包中源IP&port发送过数据，应答包才会被转发到内部主机上。Port-Restricted Cone NAT, Symmetric NAT。

Two-Way NAT

NAT内部的私有地址映射为外部的公网地址。PAT (Port address translation)解决了多台主机同时使用同一个源端口号建立不同的外部连接时的冲突问题。

STUN协议

RFC3498 STUN - Simple Traversal of User Datagram Protocol (UDP) Through Network Address Translators (NATs) (2003)
RFC5389 Session Traversal Utilities for NAT (STUN) (2008)
RFC7635 Session Traversal Utilities for NAT (STUN) Extension for Third-Party Authorization (2015)

STUN适用于锥型NAT：full cone NAT, restricted cone NAT, and port restricted cone NAT。（不适用 symmetric NAT）。

STUN是client-server协议，client和server之间可能存在一个或多个NATs，请求有两种类型:

  • Binding Requests, sent over UDP;
  • Shared Secret Requests, sent over TLS over TCP.

Server端部署在公网上，收到的请求中源地址是最靠近Server的NAT分配的，应支持如下四种应答：
(A1, P1), (A2, P1), (A1, P2), and (A2, P2)。
其中(A1, P1)是主地址和端口，端口默认3478。A2和P2可以是任意的值。

Client端发起请求：

  1. 区分锥型还是对称型：
    分别往A1和A2发送请求，如果这两次应答中MAPPED-ADDRESS一致，
    说明同一内部主机联系外部不同主机使用了同一个洞，是锥型NAT。否则，是对称型NAT。
  2. 区分具体锥型：
    往(A1,P1)发送三次请求：x、y、z，其中x无附加属性，y要求从(A2,P2)应答，z要求从(A1,P2)应答。
    首先能收到x的应答，说明存在锥型NAT。否则，存在防火墙（应答包被防火墙拦截了）；
    其次能收到y的应答，说明存在全锥型NAT。否则，不存在全锥型NAT，需要继续观察z的应答；
    最后能收到z的应答，说明存在限制锥型NAT。否则，存在端口限制锥型NAT。

Firewall通常规则：

  1. 只允许从内部主机发起会话；
  2. 按照一定规则过滤外部主机应答的数据包。
    • 应答包的源IP必须是请求包的目的IP；
    • 应答包的源IP&Port必须是请求包的目的IP&Port（此时也可称为symmetric Firewall）；

示例：端口限制锥型NAT（Endpoint-Independent Mapping & Address and Port-Dependent Filtering）

  • x:无附加
  • y:改变IP和Port
  • z:只改变Port
  • cone:发往不同IP

P2P穿越NAT

RFC5128 State of Peer-to-Peer (P2P) Communication across Network Address Translators (NATs) (2008)

TRUN协议

RFC5766 Traversal Using Relays around NAT (TURN): Relay Extensions to Session Traversal Utilities for NAT (STUN) (2010)
RFC7065 Traversal Using Relays around NAT (TURN) Uniform Resource Identifiers (2013)
RFC8656 Traversal Using Relays around NAT (TURN): Relay Extensions to Session Traversal Utilities for NAT (STUN) (2020)

TRUN适用于解决如下NAT场景：

  1. 对称型（symmetric）NAT。
  2. 通信两端（TURN/client和peer）都在NAT内时，使用中继（TURN/server）在两端之间互相转发流量。

TURN缺点：

  1. 通讯性能会受限于TRUN/Server的处理能力和带宽；
  2. 增加了通信延迟。

TURN模型：

TURN/Client --NAT-- TURN/Server(XOR-RELAYED-ADDRESS) -- PeerA/B/..
TURN/Client --NAT-- TURN/Server(XOR-RELAYED-ADDRESS) --NAT-- PeerA/B/..
TURN/client to TURN/server TURN/server to peer
UDP UDP
TCP UDP
TLS-over-TCP UDP
DTLS-over-UDP UDP
  1. 分配中继地址

TURN/Client向TURN/Server申请分配器（Allocation），申请成功后需要定期保活，一个Allocation可以同时与多个peer交换数据，分配器与中继地址（XOR-RELAYED-ADDRESS）一一对应。

  1. Send机制
    每个分配器（allocation）可以有多个授权（permission），每个授权包含一个peer的IP以及生存时间（lifetime，固定5min后过期），CreatePermission或ChannelBind会刷新授权，另外不允许主动删除授权。TURN权限借鉴了NAT的address-restricted filtering的机制，当TURN/Server收到peer应答的数据时，只有源IP地址匹配到了授权，数据才会中继给TRUN/Client。

  • Send&Data
    Send方法用于将数据从client发送到server（PeerB没有授权，数据被丢弃）；
    Data方法用于将数据从server发送到client（PeerB没有授权，数据被丢弃）。

ICE规范

RFC8445 Interactive Connectivity Establishment (ICE): A Protocol for Network Address Translator (NAT) Traversal (2018)

freeswitch

  • 场景A: 话机运行在内网，穿过NAT与公网上的FreeSwitch（运营VoIP）对接。

           话机:192.168.0.2 --NAT:1.2.3.4-- Internet -- 1.2.3.5:FreeSwitch -- 外部网关

    • 话机向FreeSwitch注册时：

           REGISTER 1000@1.2.3.5 SIP/2.0
     Contact: 1000@192.168.0.2:5060

      此时，FreeSwitch记住了Contact地址，后续呼叫1000的INVITE请求发往192.168.0.2，因为存在NAT会导致呼不通问题。

      解决方式：

      1. 话机先通过STUN服务获取到被NAT设备映射完的外网地址：UDP 1.2.3.4:5060，然后在注册请求中直接携带外网地址。
      2. FreeSwitch发现注册请求实际从外网地址：UDP 1.2.3.4:5060发出，后续呼叫发往这个外网地址上。配置<param name="apply-nat-acl" value="nat.auto">后，FreeSwitch会判断Contact地址为RFC1918规定的私网地址时，将其自动替换为SIP包的来源地址。

    • 话机向FreeSwitch注册时（Via字段携带rport字段，需话机支持sip/rport功能）：

           REGISTER 1000@1.2.3.5 SIP/2.0
     Via: SIP/2.0/UDP 192.168.0.2:5060;branch=z9d851;rport
     Contact: <sip:1000@192.168.0.2:5060;rinstance=6622cfa07990b53c>

      FreeSwitch应答时，会将应答包发往SIP请求的实际来源地址，并将实际来源端口设置为Via头部的rport的值，同时增加参数received=实际来源IP：

           SIP/2.0 200 OK
     Via: SIP/2.0/UDP 192.168.0.2:56020;branch=z9d851;rport=5060;received=1.2.3.4

      SIP客户端收到应答后，即知道了自己的外网地址，后续在重发的注册信息中，Contact地址直接填外网地址。

    • 话机呼叫FreeSwitch时：

           INVITE 9196@1.2.3.5 SIP/2.0
     Content-Type: application/sdp

     c=IN IP4 192.168.0.2
     m=audio 50452 RTP/AVP 8 0 101

      此时，因为存在NAT会导致FreeSwitch无法向192.168.0.2发送RTP包。

      解决方式：

      1. 话机先通过STUN服务获取到RTP流被NAT设备映射完的外网地址：UDP 1.2.3.4:50452，然后在sdp信息中直接携带外网地址。
      2. FreeSwitch在收到第一个RTP包后，记住RTP包实际从外网地址：UDP 1.2.3.4:50452发出，后续所有RTP包后发往这个外网地址上。

  • 场景B：话机和FreeSwitch运行在内网（公司内部IP-PBX），穿过NAT与公网上的设备对接。

       话机 -- FreeSwitch --NAT-- Internet -- 外部网关

    FreeSwitch支持通过uPnP或NAT-PMP协议在路由器上打洞，打洞完成后就知道将要映射的外网地址了。

       <param name="ext-sip-ip" value="auto-nat"/>
   <param name="ext-rtp-ip" value="auto-nat"/>

    FreeSwitch也支持通过STUN服务器获取外网地址：

       <param name="ext-sip-ip" value="stun:stun.freeswitch.org"/>
   <param name="ext-rtp-ip" value="stun:stun.freeswitch.org"/>
